## Проблемы

- Файлы PNG сохранены как RGB без alpha/transparency.
- Прозрачность только имитирована шахматной подложкой; подложка является частью изображения.
- У всех файлов одинаковый canvas size `1254x1254`, но персонаж занимает только часть canvas.
- Есть заметные пустые поля вокруг персонажа, из-за чего sprite bounds будут больше визуального тела медвежонка.
- Нет готовой последовательности кадров ходьбы с единым силуэтом, pivot point и canvas policy.
- Нет отдельного чистого кадра прыжка.
- Нет отдельного чистого кадра сидения без фона.
- Есть несколько поз, но они не оформлены как animation state set.
- Текущий первый уровень загружает `assets/images/bear_cub/bear_cub_base.png` и обрезает его source rect в коде; это рабочий обход, но не полноценный sprite pipeline.

## Подходит ли для Flame SpriteComponent

Для первого безопасного подключения после обработки лучше всего подходят:

1. `assets/images/bear_cub/bear_cub_base_2.png` - фронтальная стоячая поза для static idle, если нужен персонаж лицом к игроку.
2. `assets/images/bear_cub/bear_cub_base_5.png` - боковая стоячая поза для side-view gameplay, если движение в уровне читается слева направо.
3. `assets/images/bear_cub/bear_cub_base_1.png` - альтернативная боковая поза, но ее нужно согласовать по направлению и масштабу с выбранным gameplay view.

До удаления фона и нормальной обрезки ни один из файлов не стоит подключать как финальный `SpriteComponent`.

## Что нужно для нормальной анимации

Минимальный набор кадров:

1. `idle` - стоит/дышит.
2. `walk_01`, `walk_02`, `walk_03`, `walk_04` - ходьба.
3. `jump` - прыжок.
4. `sit` - присел/сидит в снегу при встрече с мудрецом.

Требования к каждому кадру:

- PNG RGBA с настоящей прозрачностью.
- Без запеченной шахматной, белой или цветной подложки.
- Единый visual style, масштаб и направление персонажа.
- Единый canvas size внутри одного animation state.
- Стабильный pivot/anchor, лучше всего в районе нижнего центра лап.
- Минимальные пустые поля, но с небольшим техническим padding для шерсти и антиалиасинга.

## Рекомендуемая структура ассетов

```text
assets/images/characters/bear_cub/source/
assets/images/characters/bear_cub/processed/
assets/images/characters/bear_cub/animations/idle/
assets/images/characters/bear_cub/animations/walk/
assets/images/characters/bear_cub/animations/jump/
assets/images/characters/bear_cub/animations/sit/
```

Назначение папок:

- `source/` - исходные необработанные изображения, включая текущие варианты с подложкой.
- `processed/` - очищенные одиночные PNG с настоящей прозрачностью.
- `animations/idle/` - кадры breathing/idle.
- `animations/walk/` - согласованная серия ходьбы.
- `animations/jump/` - кадр или серия прыжка.
- `animations/sit/` - кадр или серия сидения.

## План следующих этапов

1. Очистить выбранные PNG от запеченной шахматной подложки и сохранить как RGBA.
2. Обрезать пустые поля вокруг персонажа, сохранив небольшой padding для шерсти.
3. Привести все кадры внутри каждого состояния к единому canvas size.
4. Выбрать один static sprite для первого безопасного подключения.
5. Подключить static sprite без изменения механики.
6. После проверки static sprite подключить animation states: `idle`, `walk`, `jump`, `sit`.

## Следующий безопасный шаг

Сначала подготовить один очищенный static sprite в новой структуре, например:

```text
assets/images/characters/bear_cub/processed/bear_cub_idle_front.png
```

После этого можно заменить текущий временный PNG на обработанный asset отдельным маленьким изменением, не трогая механику уровня.
